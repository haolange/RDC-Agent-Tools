0001: RenderDoc MCP 工具集
0002: 遵照“为 Agent 提供与人类专家通过 GUI 操作等价的 API”这一核心思想，本报告对 RenderDoc 的全部 UI 功能进行了地毯式分析与映射，旨在构建一套真正全面、强大且具备第一性原理的 MCP (Model Context Protocol) 工具集，确保 AI Agent 能够复现人类专家在 RenderDoc UI 上的任何调试操作。
0003: 一，设计哲学
0004: 1.1，目标
0005: 本工具集的设计旨在实现两大核心目标，以赋能 Agent 在 RenderDoc 生态系统中的调试能力：
0006: 能力对齐 RenderDoc GUI：确保 Agent 无需模拟图形用户界面 (GUI) 的鼠标操作，即可获取与 GUI 面板等效的 state、resource、history 和 debug 数据及其导出物。
0007: 补齐人类专家的“组合技”：鉴于 GUI 仅提供可视化信息，而人类专家常依赖固定的排查流程（例如：pixel history → state diff → resource lifetime → shader hotfix），本工具集在原子工具之上增设了 Diagnostics 和 Macro Workflows 两层工具。此举旨在为 Agent 提供可复用、可解释且可验证的排查机制。
0008: 1.2，原则
0009: 本工具集遵循以下五项关键设计原则，以确保其高效性、可维护性和可扩展性：
0010: 显式句柄管理 (Explicit Handles)：所有核心对象（如 capture file、replay session、remote connection、shader debug session）均通过其对应的 *_id 进行显式引用，从而避免了隐式全局状态可能导致的混淆和错误。
0011: 大数据一律走 Artifact (Files as Artifacts)：对于texture、buffer、mesh、shader binary 等大型数据对象，默认将其导出为文件并返回相应的 *_path。得益于Native Multi Modality，此策略有效避免了 JSON 响应过大，提升了数据传输效率。
0012: 原子化 + 正交性 (Primitives + Orthogonality)：基础工具被设计为尽可能原子化，以提供高度的灵活性。同时，宏工具在输出结果的同时提供 trace（内部关键步骤），确保操作的可审计性和可复现性。
0013: 跨 API 的“共同语义”优先：在不同图形 API（如 D3D12/Vulkan）之间，同名工具力求返回统一的语义。特定于 API 的字段则被封装在 api_specific 子字段中，以保持接口的简洁性和一致性。
0014: “能解释”是第一等公民：Diagnostics 和 Macro 工具的输出不仅提供数据，更重要的是提供 why（结构化证据和建议的下一步行动），显著降低了 Agent 做出错误判断的概率。

0052: 2.2，返回值通用约定
0053: 除特别说明外，所有工具的返回值均包含以下字段：
0054: success (bool)：指示操作是否成功完成。
0055: error_message (str, optional)：若操作失败，提供详细的失败原因。
0056: 当返回的数据量可能非常大时，工具将优先返回 *_path（即 artifact 文件的路径），而非直接在响应中内联数据，以优化性能和传输效率。
